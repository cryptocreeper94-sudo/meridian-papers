Abstract

The six preceding papers in this series specify Meridian as a network protocol stack, a biological organism analog, an internet‑scale energy routing standard, a general theory of deterministic infrastructure, a formal security framework, and a unified energy‑data fabric. None of them answer the question a building owner, architect, or facilities manager asks first: how does this actually get installed in a real building?

Real buildings are not free‑space volumes with nodes floating in ideal positions. They are layered collections of floors, walls, ceilings, open staircases, atriums, elevator shafts, mezzanines, balconies, mechanical chases, and structural columns — each with different RF propagation properties, occupancy characteristics, and implications for where a directed energy beam may safely travel.

I introduce the Architectural Voxel Map (AVM) — a formal 3D volumetric representation of a building’s physical geometry imported into the Meridian routing engine at commissioning and enforced as a hard constraint on every beam path computation. The AVM classifies every voxel in the building volume as one of seven types, and the DRMA routing engine rejects any beam path whose geometric projection intersects a prohibited voxel class.

I define the ceiling‑plane routing model as the standard deployment posture for occupied buildings. I characterize the five void types requiring explicit routing detours — open staircases, atriums, elevator shafts, balcony overhangs, and occupied mezzanines — and specify the floor‑transition gateway architecture enabling inter‑floor energy transfer without beam traversal of occupied spaces.

I provide the formal safety proof for multi‑story deployment: that the combined enforcement of AVM constraints, ceiling‑plane routing, floor‑transition gateways, and the Biological Safety Layer makes physical harm from beam traversal impossible under all non‑destructive conditions. I close with the BIM integration pathway for native architectural design software integration.

1 Introduction

1.1 The Gap This Paper Fills

The Meridian architecture [Meridian Architecture, 2026] specifies how energy is routed through a mesh of addressed nodes. It defines the routing protocol, the control runtime, the security framework, and the self‑healing mechanisms. It does not specify where, physically, the nodes go — or how the building’s geometry constrains where the beams between them may travel.

This gap is not academic. It is the difference between a paper that describes a correct routing protocol for a free‑space volume and a product that can be installed in a hospital, an office tower, a residential building, or a warehouse without injuring anyone.

1.2 The Ceiling‑Plane Insight

The motivating insight is simple: a building’s ceiling is a surface that is, under normal occupancy conditions, above every person in the building. A beam that travels along the ceiling plane — horizontally at ceiling height between relay nodes mounted at the ceiling — passes through a geometric zone that people do not occupy. The beam path is safe not because a sensor detected no person in its way, but because no person can be in its way given the physical geometry.

This reframes the safety model from reactive (detect a person, stop the beam) to proactive (route the beam where no person can be). The BSL remains as the enforcement backstop. But the primary safety mechanism is architectural routing — choosing beam paths that are geometrically separated from occupied space.

1.3 Contributions

  1. The formal definition of the Architectural Voxel Map (AVM) — a seven‑class 3D volumetric representation of building geometry enforced as a hard constraint on every beam path.

  3. Characterization of five void types (open staircase, atrium, elevator shaft, balcony overhang, occupied mezzanine) and the specific routing detour each requires.
  4. The floor‑transition gateway architecture — the physical and protocol specification for inter‑floor energy transfer at designated transition points.

2 Background: The Gap Between Network Topology and Physical Architecture

2.1 How Current Routing Models Represent Space

The DRMA routing engine [Meridian Architecture, 2026 §9] maintains a routing table that represents the mesh as a graph: nodes as vertices, wireless links as edges, link quality as edge weights. This representation is accurate for connectivity but does not model the physical geometry of the space those links traverse. A link between two ceiling‑mounted nodes may cross over an open staircase void, meaning the beam has zero effective clearance because the floor is not there.

Routing tables are correct about connectivity; they are silent about the physical geometry of beam paths.

2.2 Why Node Positions Alone Are Insufficient

First, link paths are not points. A link between two nodes defines a beam cross‑section sweeping through a volume that may intersect an architectural void even if both endpoints are safe.

Second, buildings change. A building commissioned in January may have its staircase opening enlarged in March. Without an updatable AVM, the routing engine operates on a stale model.

Third, height alone is not the safety criterion. A node at 3.0m on floor 2, directly above an open staircase connecting floors 1 and 2, is potentially 6.0m+ above floor 1. Height is floor‑relative, not building‑relative.

2.3 The AVM as the Necessary Extension

The AVM augments the node routing graph with the physical geometry the graph cannot represent. The routing engine uses both: the node graph to find candidate paths, and the AVM to validate that candidate paths are geometrically safe before authorizing them.

3 The Architectural Voxel Map (AVM)

3.1 Voxel Grid Definition

The AVM divides the building volume into a uniform 3D grid of voxels, each a cube of side length Vsize. For standard commercial deployment, Vsize = 25 cm is recommended — sufficient to resolve staircase openings, balcony edges, and mechanical penetrations.

AVM coordinate system:
  Origin: building datum point (typically ground floor SW corner)
  X: East (meters)
  Y: North (meters)
  Z: Up (meters)
Voxel index (i, j, k) corresponds to physical volume:
  X ∈ [i × V_size, (i+1) × V_size]
  Y ∈ [j × V_size, (j+1) × V_size]
  Z ∈ [k × V_size, (k+1) × V_size]

For a typical 10‑story office building (50m × 40m footprint, 40m height), the AVM contains approximately 5,120,000 voxels. At 1 byte per voxel, the full AVM is approximately 5 MB — trivial to store and load at commissioning.

3.3 AVM Construction

Source 1 — BIM export (preferred). The Meridian BIM plugin generates the AVM automatically from the architectural model, classifying each voxel by querying the BIM model’s spatial elements.

Source 2 — As‑built survey. LiDAR point cloud scanning produces a 3D model classified into AVM voxel types by the commissioning tool.

Source 3 — Floor plan annotation. 2D floor plans annotated with void types and extruded vertically. Least accurate method, requires careful annotation.

3.4 AVM Updates

The AVM must be updated whenever building geometry changes. The commissioning tool maintains version history and requires re‑validation of all existing beam paths against the new AVM. Invalid paths are recomputed before resuming operation.

4 Ceiling‑Plane Routing: The Standard Deployment Posture

4.1 Definition

Ceiling‑plane routing requires relay nodes to be mounted at ceiling height within FLOOR_SAFE voxels. Beam paths remain within FLOOR_SAFE voxels except for the controlled final‑hop descent to destination devices.

Ceiling-plane route ρ from source s to relay chain [r&sub1;, r&sub2;, ..., r_n] to device d:
  For all hops h_i from node n_i to n_(i+1), where n_(i+1) ≠ d:
    all voxels in swept_volume(beam(n_i, n_(i+1))) ∈ FLOOR_SAFE ∪ SOLID
  For the final hop from r_n to d:
    horizontal component ∈ FLOOR_SAFE
    vertical descent component ∈ FLOOR_OCCUPIED
    final-hop protocol active (BSL engaged, tracking active, power limited)

4.2 Node Placement Requirements

Mounting height: Node center at z ≥ zsafe(floor). Typical: zceiling − 5 cm (flush) or zceiling − 15 cm (pendant).

Mounting position: Not directly above any VOID class, even if height is within FLOOR_SAFE.

Beam angle envelope: θmax = arctan((znode − zsafe) / dmin). Typical: znode = 2.7m, zsafe = 2.2m, dmin = 1.5m → θmax ≈ 18.4°.

Density: Sufficient to maintain FLOOR_SAFE paths around all VOID voxels.

4.3 Final‑Hop Protocol

The final hop — from the last relay node down to the destination device — is the only segment through FLOOR_OCCUPIED space. It is governed by:

  • Power reduction: Pfinal ≤ Prelay × attenuation_factor for the final hop distance.
  • Beam width reduction: Maximum array focusing — minimum swept volume through FLOOR_OCCUPIED.
  • Active tracking: Lume‑X continuous position tracking; beam held if device moves >10 cm from last confirmed position.
  • Enhanced BSL: Five‑modality consensus required (vs. three‑modality for relay hops).
  • Burst duration limiting: Tburst_max = 500ms, with 50ms hold between bursts for BSL sensor updates.

5 The Five Void Types and Their Routing Detours

5.1 Open Staircase

A floor opening connecting two or more levels with treads exposed to surrounding space. The void extends from the lowest connected floor to the ceiling of the highest. A horizontal beam at ceiling height traveling toward the opening has a downward projection that passes through the void, where persons on the stairs may be at any height.

AVM classification: All voxels within the staircase footprint, from z = floorN‑1 to z = ceilingN, classified VOID_STAIRCASE. 50 cm buffer zone around the perimeter.

Routing detour: Horizontal circumnavigation at ceiling level around the VOID_STAIRCASE column. For a typical 1.2m wide staircase, the detour adds one or two relay hops.

5.2 Atrium

A multi‑story open volume where multiple floor levels have been removed, spanning from ground floor to skylight. A beam departing horizontally from a node near the atrium edge passes into the volume where persons may be standing on balconies at any floor level.

AVM classification: All voxels within the atrium footprint and 50 cm buffer, from z = ground to z = roof, classified VOID_ATRIUM regardless of floor level.

Routing detour: Perimeter routing strategy — relay nodes around the atrium perimeter at ceiling height on each floor, forming a ring. Energy never crosses the atrium volume; it travels around it.

Balcony sub‑case: A balcony overlooking the atrium is handled by the VOID_ATRIUM boundary — any beam directed toward the atrium is blocked before reaching the balcony zone.

5.3 Elevator Shaft

An enclosed vertical void containing elevator cars, counterweights, and guide rails. Shaft walls are concrete or metal, providing effective RF attenuation at mmWave frequencies. The routing problem arises at shaft doors: an open door creates a temporary aperture connecting VOID_SHAFT to the adjacent corridor.

AVM classification: Shaft interior classified VOID_SHAFT. Floor‑level area in front of each door classified FLOOR_OCCUPIED.

Dynamic handling: Door‑open events extend VOID_SHAFT into the lobby temporarily via the dynamic occupancy overlay (Section 7).

5.4 Balcony Overhang

A floor extension protruding beyond the main floor plate, creating reduced effective clearance below. Persons on the balcony above are at non‑standard height — higher than floor N’s occupied zone, lower than floor N+1’s safe zone.

AVM classification: Balcony slab underside classified SOLID at actual height. Zone between slab underside and main ceiling re‑classified based on actual clearance. If clearance < 2.2m, classified FLOOR_OCCUPIED even if absolute height would normally be FLOOR_SAFE.

Routing detour: Relay nodes must not be positioned under overhangs with reduced clearance. Routing engine routes around affected zones.

5.5 Occupied Mezzanine

An intermediate floor level covering a portion of the main floor area. A relay node at the main ceiling (e.g., 6.0m) may have only 1.0m clearance above the mezzanine deck — well below zsafe — with persons directly in the beam path.

AVM classification: Mezzanine deck classified SOLID. Zone above deck to zsafe above mezzanine classified FLOOR_OCCUPIED. Zone above zsafe above mezzanine classified FLOOR_SAFE. Zone below deck uses main floor classification. This produces a layered classification correctly representing multi‑level occupancy.

Routing detour: Horizontal ceiling‑plane routes above mezzanine must be at mezzanine_height + 2.2m or above. If main ceiling is insufficient, the routing engine falls through to a perimeter route around the mezzanine footprint.

6 Floor‑Transition Gateway Architecture

6.1 The Inter‑Floor Problem

Energy cannot be beamed through a concrete floor slab at useful power levels. At 60 GHz, attenuation through a standard 20 cm concrete slab exceeds 40 dB — effectively a total block. Three inter‑floor transfer mechanisms are available:

Mechanism 1 — Open penetration. An existing aperture through the floor slab (conduit chase, pipe sleeve, or purpose‑built RF window). A gateway node pair communicates through the aperture.

Mechanism 2 — Enclosed stairwell. A continuous air volume connecting floor levels, bounded by solid walls. Gateway nodes at the top and bottom communicate through the stairwell volume. The most commonly available path in commercial buildings.

Mechanism 3 — Dedicated RF conduit. A purpose‑installed waveguide connecting gateway nodes on adjacent floors. Highest RF efficiency and simplest safety model, but requires planned installation.

6.2 Gateway Node Specification

A gateway relay node (GRN) extends the standard relay node specification with:

  • Dual‑face antenna array: Two phased array faces — one for the floor above, one for the floor below. Each face participates in its floor’s ceiling‑plane mesh.
  • Intermediate supercapacitor bank: 10× standard relay capacity for asynchronous inter‑floor energy buffering.
  • Directional isolation: Physical ground plane between faces prevents coupling.
  • Shared identity: Single Trust Layer identity with GATEWAY capability profile, visible to both floor meshes.

6.3 Gateway Placement Rules

  • Continuous air path: Clear RF path between upper and lower faces. Minimum aperture: 30 cm at 24 GHz, 15 cm at 60 GHz.
  • Accessible for maintenance: Enclosed stairwells, mechanical rooms, or above accessible ceiling tiles.
  • Within 2m of the floor transition: Minimizes beam path in the transition zone.
  • Clear of occupancy during transfer: Transfer volume must be unoccupied by design or BSL‑monitored by both GRNs.

6.4 Multi‑Floor Routing Example

Device on floor 5 receiving energy from source on floor 2:
  Source (floor 2, position A)
  → ceiling relay nodes on floor 2 ceiling plane
  → GRN-2/3 upper face (floor 2 ceiling zone)
  → inter-floor transfer (enclosed stairwell)
  → GRN-2/3 lower face (floor 3 ceiling zone)
  → ceiling relay nodes on floor 3
  → GRN-3/4 → transfer → GRN-3/4 (floor 4)
  → GRN-4/5 → transfer → GRN-4/5 (floor 5)
  → ceiling relay nodes on floor 5
  → final-hop descent to device

Each intra‑floor segment is ceiling‑plane routing in FLOOR_SAFE voxels. Each inter‑floor segment is through an enclosed stairwell or designated aperture. No beam traverses VOID or FLOOR_OCCUPIED voxels outside the controlled final‑hop geometry.

7 Dynamic Occupancy Overlay

7.1 Static vs. Dynamic Constraints

The AVM encodes static physical geometry. Dynamic occupancy constraints layer on top: a person standing in a corridor, a maintenance technician on a scaffold, a crowd in a normally sparse atrium floor. The AVM says “this corridor is FLOOR_OCCUPIED”; the dynamic overlay says “and there is currently a person at position (12.3m, 8.1m, 4.2m).”

7.2 Occupancy Sensor Integration

BSL sensors serve dual function: safety monitoring and dynamic occupancy mapping. Each relay node reports BSL sensor readings at 73 Hz through LUME_X_TELEMETRY, including biological_presence flag and estimated position. The routing engine maintains a dynamic occupancy map at AVM voxel resolution. Beam paths through dynamically OCCUPIED voxels are held until the voxel clears.

7.3 Occupancy Map Latency

Update latency is bounded by the Lume‑X control cycle: 13.7 ms maximum. A person walking at 1.5 m/s travels 2 cm in 13.7 ms — less than one AVM voxel (25 cm). The occupancy map is always current to within one voxel. The BSL hardware cutoff (independent of the control cycle, response time ≤1 ms) further reduces any exposure window to physiologically insignificant levels.

7.4 Predictive Routing

For buildings with established occupancy patterns, historical data informs routing preferences as a soft cost in the routing function. Predictive routing does not override real‑time detection — AVM constraints and dynamic occupancy map are both hard constraints.

8 The Formal Multi‑Story Safety Proof

8.1 Claim

Safety Claim: The combined enforcement of AVM constraint enforcement, ceiling‑plane routing with zsafe, final‑hop protocol, and BSL hardware cutoff makes physical harm from Meridian beam traversal impossible in a compliant multi‑story deployment under all non‑destructive conditions.

Non‑destructive conditions: No building structural element physically removed or damaged, no BSL sensor hardware physically severed or destroyed, no AVM unilaterally overridden without commissioning authority.

8.2 Proof by Case Analysis

Case 1 — Horizontal beam through FLOOR_SAFE: By AVM constraint, beam paths in FLOOR_SAFE are at z ≥ zsafe (≥2.2m above slab). No standing or seated person occupies this zone. Ceiling maintenance triggers dynamic occupancy marking. The horizontal ceiling‑plane path is safe.

Case 2 — Final‑hop through FLOOR_OCCUPIED: Five‑modality BSL consensus required before each burst. BSL hardware cutoff response ≤1 ms. At final‑hop power levels, 1 ms exposure is physiologically insignificant (below ICNIRP limits by design margin).

Case 3 — Void traversal attempt: Rejected by the beam path validation algorithm before authorization. Guardian Security’s pre‑transmission route integrity check validates against the AVM. BSL provides the final backstop if both are simultaneously bypassed.

Case 4 — Staircase/atrium perimeter grazing: The 50 cm buffer zone ensures beam center lines stay ≥50 cm from any VOID boundary. At ≤18° beam angle, the beam is physically incapable of grazing the VOID perimeter.

Case 5 — Dynamic occupancy incursion: Maximum incursion window: 1 ms at BSL hardware cutoff response time, not 13.7 ms at control cycle latency.

Conclusion: In all five cases, biological contact is prevented by at least two independent mechanisms. No single mechanism failure produces physical harm. This is the defense‑in‑depth proof.

9.2 The Ceiling‑Crawl Case for mmWave

At 60 GHz, a 10 cm phased array produces a beam approximately 25 cm wide at 3m distance — fitting entirely within FLOOR_SAFE with substantial margin. The concrete floor slab completely contains upward scatter. The floor below completely contains downward scatter from ceiling relay hops.

Recommendation: 60 GHz as primary frequency for intra‑floor ceiling‑plane routing and inter‑floor gateway transfer. 5.8 GHz secondary channel for high‑ceiling industrial spaces where 60 GHz range is insufficient.

10 Building Type Scenarios

10.1 Commercial Office Tower

Repeating open‑plan floors with suspended ceilings, central core with enclosed stairwells, 3.5–4.0m floor heights. Clean AVM profile. Relay nodes in suspended ceiling grid at 3–5m spacing (~1 relay per 15–20 m²). GRNs at stairwell landings. SSD devices fully viable throughout entire floor plate. Primary challenge: source node power connection to building electrical.

10.2 Hospital

Mix of private rooms, open wards, corridors, operating theaters. Multiple ceiling heights (2.5–4.5m). Relay nodes rated for hospital environments (antimicrobial housing, flush‑ceiling mount). CRITICAL priority class for life‑critical devices. Strongest use case: eliminating battery replacement for patient monitors (infection control benefit). Primary challenge: RF compatibility with existing medical telemetry — 60 GHz has minimal interference with 400 MHz / 900 MHz / 2.4 GHz medical bands.

10.3 Residential Mid‑Rise

Stacked residential units with 2.4–2.7m ceiling heights. Per‑unit mesh with source node plugged into standard outlet. SSD devices (thermostat, sensors, door lock, voice assistant) receive continuous power. Primary challenge: 2.4m ceiling leaves only 0.2m FLOOR_SAFE margin — requires precise node mounting and beam angle control. Phase 2 deployment target pending experimental validation.

10.4 Industrial Warehouse

Single large open volume, 8–15m ceiling height. Enormous FLOOR_SAFE zone (6–13m). Fewest relay nodes needed. Source nodes at roof structure. Primary use: powering dense IoT sensor networks without battery management. Secondary: forklift battery supplementation. Primary challenge: fast‑moving mobile equipment requires supplemental short‑range position sensors for sub‑cycle occupancy tracking.

11 BIM Integration Pathway

11.1 BIM as the Design Environment for Meridian

Meridian deployment should be designed within BIM for the same reason HVAC and electrical systems are: it is a building system with spatial, clearance, penetration, and coordination requirements. Designing Meridian outside BIM and retrofitting produces coordination conflicts that are far more expensive to resolve in construction than in design.

11.2 The Meridian BIM Plugin

Compatible with Autodesk Revit, Bentley AECOsim, and IFC‑compliant platforms:

  • AVM generation: Automatic voxel classification from the building model’s spatial elements.
  • Node placement tool: Real‑time validation against AVM and MEP clearance requirements. Highlights conflicts and suggests compliant alternatives.
  • Coverage simulation: Simulates coverage using AVM and 60 GHz beam propagation model.
  • Gateway aperture coordination: Identifies available penetration points and flags structural conflicts.
  • BIM export to AVM format: Versioned and cryptographically signed export for the commissioning tool.
  • Coordination model: All Meridian elements exported as IFC entities for standard BIM coordination workflows.

11.3 The Retrofit Pathway

For buildings without BIM: LiDAR scan (≤2 cm accuracy) → auto‑classification → technician review → AVM validation against physical walk‑through → signed AVM export.

12 Commissioning Protocol

12.1 Pre‑Installation Checklist

  • AVM complete and signed (BIM export or LiDAR survey)
  • All void classifications verified by physical walk‑through
  • Gateway aperture positions cleared with structural engineer
  • Relay node positions validated against AVM and MEP conflict check
  • Source node power connection points coordinated with electrical
  • RF interference pre‑survey completed
  • BSL sensor coverage map generated (no gaps in final‑hop zones)
  • Trust Layer identity credentials ordered for all nodes
  • Guardian Security commissioning authority credential issued

12.2 Installation Sequence

  1. Gateway relay nodes first. GRNs at all floor‑transition points before intra‑floor nodes — they are the architectural anchor points.
  2. Source nodes second. Connect to building electrical. Verify power input, supercapacitor charge, AVM import, Trust Layer registration.
  3. Relay nodes third. Install working outward from source and gateway nodes. Each auto‑announces; routing engine validates position against AVM.
  4. Coverage verification. Verify compliant routing paths to all planned device positions. Flag coverage gaps.
  5. BSL calibration. Calibrate sensor suite for specific installation geometry at each relay node.
  6. AVM live import. Import signed AVM into operational routing engine. Verify VOID path rejection.
  7. Safety validation. Confirm dynamic occupancy marking within one control cycle and BSL hardware cutoff within specification.

12.3 Change Management

Any physical change affecting the AVM requires: AVM update → re‑validation of all existing beam paths → re‑commissioning of invalid paths before the building change occurs. The commissioning tool enforces this sequence with commissioning authority signature requirement.

14 Limitations and Honest Boundaries

The AVM assumes rigid geometry. Flexible architectural elements (retractable partitions, deployable barriers) require frequent AVM updates or conservative classification as FLOOR_OCCUPIED.

Void classification at irregular boundaries is imprecise. The 25 cm voxel resolution approximates diagonal staircase edges and curved atrium walls. The conservative classification (partial overlap → void type) is intentional.

The safety proof assumes correct AVM construction. A misclassified VOID_STAIRCASE as FLOOR_SAFE invalidates the proof for that zone. The commissioning walk‑through and BSL backstop are the mitigations.

60 GHz range limitations constrain large spaces. Atmospheric attenuation may limit relay hops to 8–10m in warehouse or atrium scenarios, requiring denser node deployment.

No experimental validation exists. Beam containment in FLOOR_SAFE zones and BSL sensor coverage are based on manufacturer specifications, not measurements in actual buildings. Phase 1 experimental work should include ceiling‑plane routing validation.

BIM integration requires vendor cooperation. The Meridian BIM plugin is specified but not implemented. Development requires BIM platform API access.

15 Conclusion

The six preceding papers define what Meridian is. This paper defines where Meridian goes.

A routing protocol that cannot be safely deployed in a real building is not a product. Real buildings have open staircases, atriums, elevator shafts, balconies, and mezzanines. Every one of these features creates a geometric condition under which a naive energy routing implementation would create beam paths through occupied space.

The Architectural Voxel Map prevents this. By encoding the building’s physical geometry as a hard constraint on every beam path computation, the AVM ensures the routing engine cannot authorize a beam path that traverses a void. Ceiling‑plane routing keeps beam energy above the occupied zone. Floor‑transition gateways solve inter‑floor transfer without occupied‑space beam traversal. The BSL provides the safety backstop.

The result is a formal safety proof: physical harm from Meridian beam traversal is impossible in a compliant multi‑story deployment under non‑destructive conditions. This proof rests on four independent mechanisms — no single failure produces harm. This is the safety standard wireless energy routing requires for deployment in occupied buildings.

The BIM integration pathway closes the loop between engineering specification and architectural practice. When Meridian deployment is designed as a native BIM layer — alongside HVAC, electrical, and structural — the coordination problems that plague retrofit installations are resolved before construction.

The ceiling‑plane routing model is, at its core, a simple idea: keep the beam where no person can be, and govern the descent to where people are with maximum care. That idea, formalized through the AVM, the gateway architecture, the dynamic occupancy overlay, and the multi‑story safety proof, is what makes Meridian a real product — not a laboratory demonstration, but an energy infrastructure system that can be installed in the buildings where people live and work.

Appendix C — AVM‑DRMA Integration: Beam Path Validation Algorithm

FUNCTION validate_path(candidate_path, AVM, occupancy_map) → VALID | REJECTED
  FOR each hop (node_i, node_j) in candidate_path:
    swept_volume ← compute_swept_volume(node_i.position, node_j.position,
                                        beam_width_at(node_i, node_j))

    FOR each voxel v in swept_volume:
      voxel_class ← AVM.classify(v)

      IF voxel_class IN {VOID_STAIRCASE, VOID_ATRIUM, VOID_SHAFT}:
        RETURN REJECTED (reason: VOID_TRAVERSAL, voxel: v, hop: (i,j))

      IF voxel_class == FLOOR_OCCUPIED:
        IF (node_i, node_j) is NOT final_hop(candidate_path):
          RETURN REJECTED (reason: NON_FINAL_OCCUPIED_TRAVERSAL, hop: (i,j))
        ELSE:
          IF NOT final_hop_protocol_satisfied(node_i, node_j):
            RETURN REJECTED (reason: FINAL_HOP_PROTOCOL_VIOLATION)

      IF occupancy_map.is_occupied(v):
        RETURN REJECTED (reason: DYNAMIC_OCCUPANCY, voxel: v, hop: (i,j))

  RETURN VALID

The validation function runs for every candidate path before beam authorization by Guardian‑E, within the Lume‑X 13.7 ms control cycle budget.

Appendix D — BIM Export Format Specification (Meridian Layer)

Meridian AVM binary format (.mav extension):

Header (64 bytes):
  magic          : 4 bytes  ("MAVM")
  version        : 2 bytes  (currently 0x0001)
  building_id    : 16 bytes (UUID)
  avm_version    : 4 bytes  (incremental version counter)
  origin_x/y/z   : 24 bytes (double precision, meters)
  voxel_size     : 4 bytes  (float, meters, default 0.25)
  grid_dims_i/j/k: 12 bytes (uint32 each)
  timestamp      : 8 bytes  (Unix epoch, milliseconds)
  author_key_id  : 8 bytes  (Trust Layer identity of BIM author)
  signature      : 64 bytes (Ed25519 signature over header + voxel data)

Voxel data:
  [grid_dims_i × grid_dims_j × grid_dims_k bytes]
  Each byte: voxel class (0–6 as specified in Appendix A)
  Order: (i=0,j=0,k=0), (i=1,j=0,k=0), ..., row-major in i, then j, then k

The signature covers the full voxel data array and the header (excluding the signature field). The commissioning tool verifies the signature against the author’s Trust Layer public key before importing any AVM.
